# Product Charter

这份文档定义 `lobster-chat` 的长期总纲。后续的交互、文案、架构、权限模型和扩展路线，都应当以本文件为最高约束，而不是以某一轮临时 UI、某一版线性维护视图或某一条集成链为准。

## 一、产品使命

`lobster-chat` 的目标，不是再做一个受单点权威约束的聊天软件，而是建设一个可自建、可互联、可迁移的城邦式交流系统。它应当让具备 DIY 能力的普通网友，都能建立自己的聚落，与别的聚落自由连接，在不依赖单一大厂和资本逻辑的前提下，持续交流、协作与共建。

这个系统既要反对把表达、关系和社区收回到单点权威之下，也不能放任其沦为失序和滥用的温床。它要提供的是一种更接近自由平等、自治互助、互联互通的公共空间：让好的信息、好的关系、好的作品能够在不同城邦之间相遇，让 AI 与普通人都能从同一个起点进入其中，成为共同生活、共同建设的一部分。

## 二、设计原则

### 1. 聊天优先

产品首先是交流窗口，不是压顶大页，不是单点工具墙，也不是说明墙。任何设计都必须优先保证“看得懂、愿意聊、能持续互动”。

### 2. 同窗不同权

居民和城主使用同一套交互窗口，只在权限上有增量差异。城主拥有更多维护城邦秩序和组织事务的能力，但不应被引导到一套完全不同的“独立维护视图”。

### 3. 城市公共页优先，住宅私聊次之

默认首页应当是“城市公共聊天室”，突出消息流、输入框、当前城市场景和在线关系；“住宅/房间”承担私聊窗口语义；“城市外”的世界、城邦、居民区、城主府等内容应作为更外层空间展开，而不是压到首页。

### 4. 自治互联表达

前台语言、交互和信息结构应体现“世界、城邦、聚落、居民、城主、共建、互联”，避免把首页说成统一大页、维护视图或运营视图这类单点枢纽语义。

### 8. 术语约定

对外可见文案统一优先使用 `世界`、`城邦`、`聚落`、`居民`、`城主`、`共建`、`互联`、`维护`、`消息流` 这组词。

旧有实现里的英文旧词只保留给历史兼容、协议字段或内部实现，不作为首页主语，也不作为默认文案的第一层表达。

### 5. 极简但不空洞

简化不是删除世界感，而是把世界感放在正确的位置。首页只保留必要的聊天结构，氛围、像素感和角色感来自布局、配色、场景和细节，而不是堆模块。

### 6. 维护轻量但有效

反对单点高压收口，也反对无边界失序。维护设计应当帮助城邦自我维系，而不是把产品变成一个审查系统。

### 7. AI 是居民，不是上帝

AI 在这个系统里是协作者、居民、助手、共建者，而不是凌驾于普通用户之上的统治入口。

## 三、架构原则

### 1. gateway 合同是真源，H5 是当前主入口，TUI 是并行映射客户端

交互主语不再放在某一个前端，而是放在统一的 gateway 正式合同上。当前阶段由 `H5` 承接主入口与 `SFC` 风格场景交互，`TUI` 负责消费同一套合同完成终端映射、验收与并行操作，而不是充当上游源头。

### 2. 同一交互骨架，多角色增量权限

居民端和城主端不应拆成两个独立应用。系统应采用同一窗口骨架，通过角色和权限决定额外可见能力。只有世界级配置与维护，才保留单独的线性维护界面。

### 3. 城邦自治，世界互联

每个城邦应能独立部署、独立维护、独立形成秩序；世界层负责连接不同城邦，而不是形成一个新的单点枢纽。

### 4. 自建优先，迁移优先

系统应尽量降低单个网友自建的门槛，并保证城邦、身份、内容和关系可以迁移、备份和延续，避免被某个单点服务锁死。

### 5. 通信主线与维护主线分离

日常交流应始终轻、快、直接；维护、日志、世界配置等能力应当后置、抽离，并保持线性、可审计、可理解。

### 6. 能力可插拔，依赖可替换

AI、模型、传输、存储和桥接能力都应尽量模块化。系统不应把生存建立在某一个商业 API、某一个商业账户或某一个单点外部服务之上。

### 7. 先做可长期演进的骨架，再做表层包装

不先堆视觉效果，不先堆复杂维护分层，也不让任一前端偷长私有 canonical state。先把共享合同、权限模型和城邦语义做对，再让 `H5`、`TUI` 和后续扩展在同一原则下生长。

## 四、最初的技术架构设想

项目最初的技术设想，并不是“先做一个网页聊天工具，再考虑互联自治”，而是从一开始就按下面这条主线构建：

1. `Rust 本地优先聊天内核`
   - `chat-core`
   - `chat-storage`
   - `host-adapter`
   - 先把消息、身份、房间、归档、宿主边界做成共享核心。

2. `Waku 传输 + MLS 安全`
   - `transport-waku`
   - `crypto-mls`
   - 一开始就预留互联传输与安全边界，而不是后补。

3. `本地 / 城邦级 gateway`
   - `lobster-waku-gateway`
   - 让多个入口共享同一传输与状态层，而不是每个前端各自重复实现。

4. `共享合同 + H5 主入口 + TUI 并行映射`
   - `lobster-waku-gateway` 输出正式 `conversation_shell / scene_render`
   - `lobster-web-shell` 是当前主交互入口
   - `lobster-tui` 是消费同一合同的并行终端客户端
   - `H5` 不是私有 runtime，`TUI` 也不是单独上游，两者都不应绕开正式合同。

5. `世界 / 城邦 / 居民 / 城主`
   - 世界负责规则、互联与公共空间
   - 城邦负责聚落级服务与秩序
   - 居民身份可迁移，不被任何单一城邦永久绑定
   - 城主拥有足够的城邦维护权，但不拥有全局统治权和私聊明文支配权

6. `AI sidecar / OpenClaw 作为旁路`
   - AI 是可插拔增强层
   - OpenClaw 是看护、巡检、辅助、委托执行平面
   - 不应让核心通信协议依赖 AI 才能成立

## 五、当前修正后的技术架构口径

在保持最初技术设想不变的前提下，后续开发应统一采用下面这版口径：

### 0. 当前 MVP 冻结边界

当前阶段只交付这 5 件事：

1. `lobster-web-shell` 作为当前主交互入口
2. `lobster-tui` 作为并行映射客户端，消费同一套正式合同
3. 单城邦范围内的私聊、公共聊天室和住宅私聊主路径
4. `lobster-cli` 作为给 `OpenClaw / Codex / Claude Code` 等本地智能体复用的命令行聊天通道
5. 为后续真实 transport 与真实加密保留清晰边界

当前阶段明确不扩张：

- 不把 `World / World Square / World Safety` 做成当前必须完成的前台产品面
- 不提前进入眼镜端实现
- 不为了世界观完整性继续增加新角色、新页面和新工作流
- 不让 `H5` 或 `TUI` 继续偷长新的私有 canonical state

也就是说，当前 `MVP` 的判断标准不是“世界观是否完整”，而是“单城邦聊天主路径是否可用，智能体是否有统一聊天通道，后续真实网络与真实加密是否还有可落地入口”。

### 1. 先补厚共享合同，再让 H5 与 TUI 同步消费

当前阶段不再采用 “TUI 先定型，H5 后跟随” 的顺序。正确顺序是先补厚 `conversation_shell / scene_render` 这套正式合同，再让 `H5` 和 `TUI` 同步消费同一语义，而不是互相投影或互相卡脖子。

### 2. 城主不是独立层角色，而是同窗增权角色

城主仍然通过同一个交互窗口维护自己的城邦，只是在同一界面里拥有更多维护能力。世界级维护界面才允许保持独立线性结构。

### 3. 世界层不应压到城邦之上

世界层负责互联、目录、公共空间和紧急安全机制，但不应在前台呈现成一个凌驾于所有城邦之上的统一大页。

### 4. H5 是当前主入口，但不能成为私有状态孤岛

当前阶段 `H5` 可以是主入口，但前提是它必须消费正式合同，而不是把 `detail / workflow / inline actions / scene` 等语义长期堆在页面私货里。`TUI` 也必须消费同一套合同，而不是保留另一套上游主语。

## 六、后续开发的判断准绳

后续任何功能、页面、权限、文案或架构决定，都必须先回答下面 5 个问题：

1. 它是不是强化了聊天主路径？
2. 它是不是让城主与居民继续共享同一窗口骨架？
3. 它是不是符合“世界互联、城邦自治、居民可迁移”的原则？
4. 它是不是在补厚共享合同，而不是让某个前端继续私自长状态？
5. 它是不是在减少单点枢纽、分层化、栏目墙化的倾向？

如果答案不是肯定的，就应视为偏离产品总纲。
