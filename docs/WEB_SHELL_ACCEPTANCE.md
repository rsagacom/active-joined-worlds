# Web Shell Acceptance

这份清单只看页面表现与交互结果，不要求先读源码。当前验收基线是 H5 像素场景 IM：`主城` 是群聊场所，`住宅` 是私聊场所，`世界入口` 是地铁候车站式导航页，`世界广场` 是独立只读公共入口。

## 需要打开的页面

- `apps/lobster-web-shell/index.html`：主城 / 群聊。
- `apps/lobster-web-shell/creative.html`：住宅 / 私聊视觉基线。
- `apps/lobster-web-shell/unified.html`：世界入口 / 地铁候车站导航。
- `apps/lobster-web-shell/world-square.html`：世界广场 / 只读公共入口。
- `apps/lobster-web-shell/user.html`：兼容跳转入口，保留 query 参数后重定向到 `creative.html`；不再独立承载旧住宅 UI。
- `apps/lobster-web-shell/admin.html`：城主治理兜底页，当前只做基础可用和移动端兜底，不作为视觉主线。

## 准备方式

1. 如果要验证真实发送、同步和导出，先启动本地 gateway。
2. 用任意静态服务器挂出 `apps/lobster-web-shell/`。
3. 需要接真实消息时，在页面 URL 后追加 `?gateway=http://127.0.0.1:8787`。
4. `creative.html` 接真实 gateway 时，未登录身份是 `访客`；页面应请求 `GET /v1/shell/state?resident_id=访客`，显示紧凑邮箱 OTP 登录片，并禁用正式发送。
5. 完成 OTP 登录后，页面应保存返回的 `resident_id`，重新请求 `GET /v1/shell/state?resident_id=<resident_id>`，并只显示该居民可见的私聊/公共会话。
6. `user.html` 不再独立承载旧住宅 UI；打开后应保留 query 参数并跳转到 `creative.html`。

最小脚本化 smoke：

```bash
cd .
./scripts/smoke-web-shell.sh
```

这条脚本当前覆盖：

- `index.html` / `creative.html` / `admin.html` / `unified.html` / `world-square.html` 的静态入口与栏目结构
- `creative.html` 的 fake-DOM 启动、默认会话选中、切换会话后输入区刷新
- `creative.html` 的 gateway 居民态请求、访客发送禁用、OTP 登录后刷新
- `user.html` 的兼容跳转行为（保留 query 参数重定向到 `creative.html`）
- `composer-state.js` 的发送可用性合同

示例：

```bash
cargo run -p lobster-waku-gateway -- --host 127.0.0.1 --port 8787
cd apps/lobster-web-shell
python3 -m http.server 8080
```

If you are using the current local preview, the same pages are also available at `http://127.0.0.1:18080/`, `http://127.0.0.1:18080/creative.html`, `http://127.0.0.1:18080/user.html`, `http://127.0.0.1:18080/admin.html`, `http://127.0.0.1:18080/unified.html`, and `http://127.0.0.1:18080/world-square.html`.

## 第一眼检查

### 1. 主城 / 群聊

- `index.html` 第一眼应读成“中国现代城区里的公共群聊”，不是说明墙、总览页或后台。
- 顶部 HUD 只保留居中标题 `MAIN CITY / 主城` 与轻状态；场景内不再放大副标题块。
- 左侧只保留 `住宅`、`主城`、`世界入口` 三个动线入口。
- 消息流按微信式左右布局：对方在左，己方在右，头像常驻，昵称、时间和气泡间距稳定。
- 底部输入区包含 `@用户名`、颜文字按钮、可拖拽高度文本框和发送按钮。颜文字面板应是上排分类 tab，不应要求用户在长列表中上下找分类。
- 点击场景空白区域应进入清屏模式，场景铺满 UI 区域；再点空白区域恢复聊天层。
- 地铁口热点应跳转 `unified.html`。

### 2. 住宅 / 私聊

- `creative.html` 第一眼应读成私人房间或工位私聊页，视觉基线与主城一致：完整像素背景、透明热点层、顶层聊天气泡。
- 顶部 HUD 只保留 `PRIVATE ROOM / 住宅` 与状态，不再显示“灵感工位”等旧副标题块。
- 私聊消息同样按左人右己，头像、昵称、气泡圆角和间距要与主城统一。
- 楼梯热点应可见且可点击，跳转 `index.html`。
- 点击空白区域应和主城一样进入宽阔清屏模式，而不是保持被 UI 框压住的窄视野。

### 3. 世界入口 / 地铁候车站

- `unified.html` 第一眼应读成“地铁候车站式世界入口”，不是聊天页或黑色传送门。
- 世界入口不展示聊天框；它是导航页。
- 点击 `世界线路图` 应弹出可选择前往的其他主城面板；当前可以先用本地样例，不要求真实跨城数据。
- 顶部、左栏、边框风格应与主城和住宅一致，不额外出现“导航”这类冗余标题。
- 点击空白区域应进入同样的清屏模式，让地铁候车站背景完整展开。

### 4. 正式居民私聊 IM

- `creative.html` 是唯一住宅 / 私聊入口，接 gateway 时必须尊重 `GET /v1/shell/state?resident_id=<id>`。
- 接真实 gateway 且未登录时，登录片应是紧凑条；未登录身份为 `访客`，正式发送被禁用。
- 完成 OTP 登录后，页面应保存返回的 `resident_id`，重新请求 shell state，并只显示该居民可见的会话。
- `user.html` 不再独立承载旧住宅 UI；打开后应保留 query 参数并跳转到 `creative.html`。

### 5. 世界广场 / 只读公共入口

- `world-square.html` 是独立世界广场入口，不再只是概念图预览；当前阶段只读展示，不接入聊天输入框、论坛发帖或治理操作。
- 如果 URL 带 `?gateway=http://127.0.0.1:8787`，页面可以只读拉取 `GET /v1/world-square` 填充摘要卡；没有 gateway 时保留本地 fallback。
- 页面应展示三类轻量摘要：`公告`、`公共讨论`、`跨城发现`。如果功能暂不可用，应显示 `只读预览` / disabled 状态，不使用空链接或无效按钮。
- 三类轻量摘要默认应收起到小的 `广场信息` 入口；用户主动展开后才显示，避免说明卡长期遮挡像素场景。
- 背景图仍可使用 `assets/pixel/concepts/world-square-concept-20260427.png` 和压缩 AVIF 版本作为视觉底图，但文字和交互必须由网页层控制。
- 世界广场不能混进 `unified.html`；`unified.html` 仍是地铁候车站式世界入口，负责线路导航和进入世界广场。

## 交互验收

### 5. 会话与时间线

- 会话列表更像聊天收件箱，不像功能抽屉。
- 当前会话有明确高亮，未读有明显提示。
- 空会话必须显示可理解的空状态，不能留白，且先引导用户选会话、发第一句。
- 空态卡片应直接说清“先选会话，再输入第一句”，而不是只给技术状态或留白。
- 时间线在发送中、待同步、发送失败、回退快照时都有可见提示。

### 6. 输入区

- 输入区文案应先说明“发到哪里”，再出现技术状态。
- 输入区和空态文案都应先说当前会话，再说同步或网关状态。
- 会话未选中时，落字区仍可见且仍可编辑，但要明确说明“先选会话后才能发送，草稿会跟着当前窗口保留”。
- `Enter` 发送、`Shift+Enter` 换行、`ArrowUp` 取回上一条可以正常工作。
- 没接 gateway 时，也要明确告诉用户消息会先保存在本地样例时间线。
- 接真实 gateway 且仍为 `访客` 时，输入区应显示“请先登录后发送”，发送按钮禁用；无 gateway 的本地预览仍可输入和发送样例消息。
- gateway 写入层也必须拒绝 `访客` 正式发送，不能只依赖前端按钮禁用。
- 窄屏下输入区仍然可见、可点、可发送，不应被侧边区域挤掉。
- 空会话时，输入区与时间线空态都应先鼓励用户选会话、发第一句，而不是解释系统状态。

### 7. 消息发送

- 至少完成一次真实发送，确认消息进入当前线程。
- 发送中按钮和状态提示会变化，成功后恢复。
- 若 gateway 回执较慢，用户仍能看到本地回显或待同步状态。

### 8. 消息来源切换

- 页面能显示当前是本地样例、当前网关还是外部消息来源。
- 接上 gateway 后，消息来源状态应与实际连接一致。
- 断开和重新连接消息来源时，无需离开当前页面。

### 9. 导出

- 可以导出当前会话记录。
- 可以导出当前可见历史。
- 导出的格式必须与选择项一致：`markdown`、`txt`、`jsonl`。

### 10. 窄屏切换

- 把浏览器缩到移动端宽度后，仍能在会话列表与消息区之间来回切换。
- 来回切换后，当前会话和当前线程不能丢失。
- 首屏仍应优先保证消息阅读和输入，而不是侧边入口。

## 明确的通过 / 不通过标准

### 通过

- 居民端、城主端、世界共建页都能读成“聊天优先”的产品页面。
- 至少一次真实发送成功，且状态反馈完整。
- 导出成功，格式正确。
- 消息来源状态可见且与实际连接一致。
- 窄屏切换后，会话、线程、输入区仍可正常工作。

### 不通过

- 第一眼更像说明墙、总览页、调试器，而不像聊天产品。
- 第一眼像“复杂总览页”而不像人愿意停留的聊天产品。
- 侧边工具区或世界入口抢走消息流和输入区的主位。
- 输入区看起来像命令栏，用户不清楚当前会发到哪里。
- 用户端或城主端仍然要靠阅读长说明，才能理解城邦/居民/城主这些对象是干什么的。
- 世界页仍然是一堵表单和杂乱栏目墙，看不出线性分类和栏目边界。
- 发送、待同步、失败、空状态中任一关键反馈缺失。
- 窄屏下无法稳定切回当前线程，或输入区被挤到难以操作。
